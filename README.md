# claude-account-switcher

Запуск Claude Code под разными аккаунтами (рабочий / личный) на одной macOS-машине
без `logout`/`login`. Общий конфиг `~/.claude` остаётся один — меняется только аккаунт.

## Зачем это нужно

На macOS Claude Code хранит OAuth-токен в Keychain под **фиксированным** именем
`Claude Code-credentials`, привязанным к системному пользователю — а не к каталогу
конфига. Из-за этого `CLAUDE_CONFIG_DIR` **не изолирует аккаунт**: `/login` под другим
аккаунтом просто перезаписывает ту же запись, и оба профиля оказываются на одном
аккаунте.

Решение — выбирать аккаунт через переменную `CLAUDE_CODE_OAUTH_TOKEN`. По приоритету
аутентификации она проверяется **раньше** Keychain, поэтому её можно задавать на лету
для конкретного запуска. Токены лежат в Keychain как отдельные записи
(`claude-oauth-<profile>`), а тонкая обёртка в zsh подставляет нужный токен только в
свой процесс — значит в разных терминалах можно держать разные аккаунты одновременно.

## Установка

1. Подключи скрипт в `~/.zshrc`:

   ```bash
   source ~/Documents/"My Projects"/claude-account-switcher/claude-switch.sh
   ```

2. Перезагрузи шелл: `source ~/.zshrc` (или открой новый терминал).

3. Сохрани токены для каждого аккаунта:

   ```bash
   claude setup-token      # войти ЛИЧНЫМ аккаунтом в браузере → скопировать sk-ant-oat-...
   cl-add personal         # вставить токен

   claude setup-token      # войти РАБОЧИМ аккаунтом → скопировать токен
   cl-add work             # вставить токен
   ```

## Использование

```bash
claude-personal           # сессия под личным аккаунтом
claude-work               # сессия под рабочим аккаунтом
cl personal --resume      # любой профиль + флаги claude прокидываются как есть
cl-list                   # список сохранённых профилей
cl-rm personal            # удалить профиль из Keychain
```

Обычный `claude` без обёртки по-прежнему ходит под дефолтным аккаунтом из `/login`
(Keychain). Можно вообще не логиниться и всегда ходить через `claude-personal` /
`claude-work`.

## Команды

| Команда | Что делает |
|---|---|
| `cl <profile> [args]` | запустить Claude Code под профилем |
| `cl-add <profile>` | сохранить токен профиля в Keychain |
| `cl-rm <profile>` | удалить профиль |
| `cl-list` | показать сохранённые профили |
| `claude-personal` / `claude-work` | алиасы для `cl personal` / `cl work` |

## Безопасность

- Токены хранятся только в macOS Keychain. В `.zshrc` и в этом репозитории их нет.
- Каждый запуск выставляет `CLAUDE_CODE_OAUTH_TOKEN` только для своего процесса —
  глобальный `/login` в Keychain не перезаписывается.

## Альтернатива: claude-swap

Есть готовая сторонняя тула — [`realiti4/claude-swap`](https://github.com/realiti4/claude-swap)
(`uv tool install claude-swap` / `pipx install claude-swap`). Она делает то же по сути:
бэкапит OAuth-токены и **подменяет** ту самую запись `Claude Code-credentials` в Keychain
при переключении. Умеет больше: добавление аккаунтов через CLI, сессионный запуск
(`cswap run 2`), авто-выбор аккаунта по остатку квоты (`--strategy best` /
`--strategy next-available`), кросс-платформенность (macOS / Linux / WSL / Windows).

### Почему в итоге делаем сами

- **Меньше движущихся частей.** Нам нужен один сценарий — «личный проект под личной
  подпиской». Для этого хватает ~30 строк zsh поверх штатных `claude setup-token` и
  `security` (Keychain). Ставить Python 3.12 + стороннюю тулу ради этого избыточно.
- **Доверие к кредам.** claude-swap получает доступ к OAuth-токенам обоих аккаунтов и
  пишет в Keychain. Для **рабочего** аккаунта это вопрос политики компании — свой
  скрипт прозрачен и не отдаёт токены никуда наружу.
- **Не нужны фичи под обход лимитов.** Стратегии `best` / `next-available` заточены под
  жонглирование множеством аккаунтов ради квоты — это не наша задача.
- **Параллельность по умолчанию.** Наша обёртка выставляет токен только в свой процесс,
  поэтому два терминала = два разных аккаунта одновременно, без переключения глобального
  состояния.

Если аккаунтов станет много и захочется готовый менеджер с переключением и стратегиями —
claude-swap разумный выбор. Для текущего сценария свой скрипт проще и безопаснее.

## Бонус: авто-выбор по папке (direnv)

Чтобы личный проект сам подхватывал личный аккаунт по `cd`, можно скрестить с
[direnv](https://direnv.net/). В корне личного проекта — `.envrc`:

```bash
export CLAUDE_CODE_OAUTH_TOKEN="$(security find-generic-password -s 'claude-oauth-personal' -a "$USER" -w)"
```

Затем `direnv allow`. Токен берётся из того же Keychain-стора. `.envrc` добавь в
`.gitignore`.
